# Public Performance Baseline Evidence Design

**Status:** Approved for direct implementation under the repository owner's standing review instruction on 2026-07-17

**Roadmap:** Phase 3 — observability and quality gates

## Decision

ORAG will publish one versioned, static evidence directory for a real execution
of the existing deterministic-mock public SDK performance baseline. The
directory will be served by the hosted documentation site and contain:

- `report.json`: the strict `orag.performance-baseline.v1` report generated by
  `oragctl benchmark-run` at the measured commit;
- `environment.json`: a deliberately small disclosure of CPU model, memory,
  operating-system/build, Go version, architecture, fixed command and capture
  timestamp; it must never contain host name, user name, IP address, tokens or
  provider credentials;
- `manifest.json`: the evidence schema, evidence ID, immutable build revision,
  report fingerprint and relative artifact names; and
- `SHA256SUMS`: SHA-256 checksums for the three JSON artifacts.

The first evidence ID is `2026-07-17-darwin-arm64-main-75eda8f`, attached to
the `75eda8f` merge commit. It is a local deterministic-mock observation, not
a production benchmark, cross-hardware comparison, cloud-network result or
provider throughput claim.

## Alternatives considered

1. **Commit a single report only.** Lowest effort, but consumers cannot assess
   the machine or independently detect modification. Rejected.
2. **Publish static evidence directory with disclosure and checksums.** Makes
   the concrete result independently inspectable with the existing static docs
   deployment and needs no new service credential. Chosen.
3. **Build a hosted benchmark database/dashboard.** Useful after multiple
   independent reports exist, but adds a service and schema before there is a
   stable public corpus. Deferred.

## Capture and verification flow

1. The capture script requires an explicit output directory and full Git
   revision. It invokes the existing public baseline runner with the pinned
   load, creates the safe environment disclosure, writes a manifest and
   checksums, then invokes the verifier before reporting success.
2. The verifier rejects absent artifacts, checksum mismatch, unexpected
   manifest keys, malformed JSON, an unapproved report, a report/build mismatch
   or disclosure keys outside the allowlist. It delegates report semantics to
   `oragctl benchmark-report`.
3. The checked-in evidence is linked from the hosted performance page and the
   operator guide. A contract test requires both the capture/verify commands
   and the honest scope language.
4. Documentation deployment exposes the exact static directory. An anonymous
   HTTPS request and the verifier provide publication evidence.

## Safety and scope

- Capture never reads model/provider secrets, `.env`, object storage settings,
  source IP, hostname or account identity.
- The runner uses `MockConfig`; reported USD cost remains zero by contract.
- Each new machine, runtime or source revision creates a distinct evidence ID;
  reports whose environment/build/load fingerprints differ are audit records,
  not comparable results.
- This does not satisfy the roadmap's production-pilot or external adoption
  exit criteria. It closes only the missing public, disclosed local baseline
  evidence.

## Acceptance evidence

- `make performance-baseline-evidence-verify` validates the checked-in
  directory.
- A focused contract test covers the static evidence, Make targets and docs.
- `make agent-gate` and `./scripts/build-docs-site.sh` pass.
- The hosted report and checksum list are anonymously readable over HTTPS.
